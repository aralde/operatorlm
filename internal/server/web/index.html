Aliases

Virtual model names that fan out across multiple real targets. Useful for cross-provider fallback, key rotation, and rate-limit pooling — like LiteLLM's model_list.

How routing works
  1. Targets are sorted by order (ascending = preferred).
  2. Targets in cooldown (after a 5xx/429) or over their rpm are skipped.
  3. The first available target is tried; on retryable failure the next is used.
  4. 4xx errors (except 429) are returned as-is — no fallback.

Reliability

            
Retry, circuit breaker, timeouts and fallback. These settings make the proxy behave like a production-grade gateway — transient upstream failures get retried with backoff, repeated failures trip a circuit, and a global fallback alias rescues exhausted requests.

          

        


        

          How a request flows
          
    
            
  1. Resolve the model to attempts (alias targets in order).
    2. 
            
  2. For each attempt: ask the breaker (closed/open/half-open) and the rpm window.
    3. 
            
  3. Run with retries — exponential backoff + full jitter (and Retry-After if upstream sets it).
    4. 
            
  4. Outcome updates the breaker: success closes; 5xx/429/network trip after N consecutive failures.
    5. 
            
  5. If everything is exhausted, the request retries through default fallback alias.
    6.

Routing rules

          
    
            
  1. If model matches an alias name → alias router (filtered by cooldown/rpm, ordered).
    2. 
            
  2. Else if model starts with a provider's prefix → direct slug routing (whitelist enforced if configured).
    3. 
            
  3. Else → 400 no provider or alias matches model "X".
